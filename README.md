# GameVault — Game Acquisition Intelligence System
## The problem

GameVault Publishing is a small, risk-averse publisher. They don't build games — they acquire *proven* ones and port them to new platforms (console, mobile). Because they're small, a single bad acquisition hurts, so they need to tell genuine, durable demand apart from a passing fad — across thousands of candidate games, using evidence rather than gut feel. Their three stated pain points were: analysis that stops at star ratings and review counts; manual review-reading that doesn't scale; and no central place to compare and track candidates. This project answers all three.

## The approach, in one idea

> **Use machine learning where the data supports it; keep the final decision transparent.**

The dataset has 14,617 labelled reviews — plenty to train a model that *reads* review text at scale. But it has only 23 games and **no historical record of which acquisitions actually succeeded**, so there is nothing to train a "good acquisition" model against. Training one anyway would mean fitting noise and handing a risk-averse client a black box they can't interrogate. So the architecture splits cleanly:

- **ML reads the reviews** (Notebook 03): a sentiment classifier and theme extraction turn unstructured text into structured signals.
- **A transparent weighted formula makes the decision** (Notebook 04): the Acquisition Score is a documented, multi-criteria recipe — like a credit score — where every point can be traced back to a cause.

## Key decisions (and why they're defensible)

- **Flag, don't delete** (Notebook 01). Problematic reviews are labelled, not removed. The clearest example: reviews from players with zero hours played look like junk, but they're 73% negative versus a 29% baseline — they're refund-and-protest votes, a real signal. Deleting them would have erased information. Filtering then happens per-task, at the point of use.
- **Discovery-led anomaly detection** (Notebook 02). Rather than assuming where problems were, a neutral test scanned every game for any quarter wildly out of line with its own normal. It surfaced GTA V's 2017 review bomb on its own — sentiment crashed to 24% positive in one quarter against a ~77% lifetime norm — proving that a single lifetime average can badly mislead, and that sentiment must be tracked over time.
- **A see-through model over a fancier one** (Notebook 03). A simple, inspectable classifier was chosen deliberately over a heavier transformer, because being able to show *why* each call was made matters more here than a couple of accuracy points. An off-the-shelf tool was tested first and scored no better than guessing (72% against a 71% base rate); the trained model reached 84% and actually caught negative reviews.
- **No ML for the final score** (Notebook 04), for the reasons above — paired with a **weight-sensitivity analysis** that shakes the weights 2,000 times and shows the top of the ranking barely moves, which answers the obvious "your weights are arbitrary" objection head-on.
- **Confidence over false precision throughout.** Five games have ≤10 reviews; they carry a Low-confidence tier and are never allowed to look as authoritative as well-evidenced games. Where a number can't be computed honestly, the system says "insufficient data" rather than guessing.

## How to run

**1. Environment** (Python 3.12)
```bash
pip install -r requirements.txt
```

**2. Add the data.** Put `steam_reviews_1.csv` in the repo root.

**3. Run the notebooks in order** — each produces the inputs for the next:
```
01_data_preparation.ipynb   →  data/reviews_prepared.csv
02_eda.ipynb                →  figures/
03_nlp_sentiment.ipynb      →  data/nlp_review_scores.csv, data/nlp_game_signals.csv, models/
04_scoring_model.ipynb      →  data/acquisition_scorecard.csv
```
(The notebooks self-install a couple of small packages if missing; seeds are fixed, so results are reproducible.)

**4. Run the dashboard:**
```bash
python dashboard/prepare_data.py      # builds the app's data file from the CSVs
cd dashboard
npm install
npm run dev
```

## Results at a glance

How much data each stage could honestly use (flag-don't-delete in action):

| Pipeline | Reviews used | Share |
|---|---|---|
| Engagement (all reviews) | 14,617 | 100% |
| Text / NLP (excludes empty & non-English) | 14,477 | 99.0% |
| Playtime (excludes zero-hour) | 14,280 | 97.7% |

Headline outcomes:

- **Reading reviews at scale:** off-the-shelf sentiment ≈ guessing (72%); the trained classifier reached **84% accuracy** and lifted detection of negative reviews substantially.
- **The ranking rewards balance, not size.** Top candidates: **Slay the Spire (91), Terraria (91), Subnautica (90), RimWorld (89), Left 4 Dead 2 (88)** — mid-sized, loved, durable. The mega-titles scored in the 60s–70s and are unrealistic targets for a small publisher anyway.
- **Two independent methods agreed.** The timing-based anomaly scan (Notebook 02) and the text-based theme extraction (Notebook 03) *both* flagged GTA V and Skyrim SE as having protest-driven negativity (≈61% of complaints aimed at publisher conduct) — negativity that may not transfer to a new owner, which the scorecard records.

## Limitations (stated honestly)

The score's weights encode judgement — mitigated by the sensitivity analysis and exposed as adjustable sliders in the dashboard, not hidden. Scores are *comparative within this 23-game catalogue*, not absolute grades. The sentiment model is word-based, so sarcasm can fool it (~1.5% of reviews, quantified in Notebook 03). The provided dataset arrived pre-cleaned, so a production deployment on raw Steam text would need re-validation. None of these affect how the signals are used: as inputs to a transparent score a human acts on, never as an autonomous verdict.
